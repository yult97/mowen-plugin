# PRD：墨问笔记助手 — 对接火山引擎流式 ASR（从 Mock 到真实链路）

> 版本：v1.0  
> 目标：在你已完成 **Mock 吐字 UI** 的基础上，对接 **火山引擎“大模型流式语音识别（WebSocket）”**，跑通「录音 → 流式转写 → Typeless 吐字/修订 → 编辑 → 保存到墨问」整条链路。  
> 约束：**不改变现有 UI/交互规范**（主色/圆角/按钮/状态机沿用）；本 PRD 聚焦 **ASR 真实接入**、数据流、风控与验收。

---

## 1. 背景与目标

### 1.1 背景
当前语音笔记功能已用 Mock 实现：
- UI 状态机（准备/录音/暂停/整理/编辑/保存/已保存）
- 吐字模型（partial/final/revise 的 UI 呈现）

下一步需要接入真实 ASR，保证：
- 低延迟吐字
- 稳定连续、不丢字
- 断线/权限/异常可恢复
- 不暴露第三方密钥（浏览器扩展安全）

### 1.2 目标（必须达成）
1) **实时流式转写**：录音开始后可持续收到 partial/final（吐字流畅）。  
2) **结果映射**：把火山输出映射到你既有 `partial/final/revise` 数据模型，UI 不重写。  
3) **安全接入**：浏览器端不直接调用火山；通过自有后端/边缘中转服务代理。  
4) **质量可控**：支持关键参数开关（如标点/ITN/热词等——以火山文档为准），并有可观测性与回放能力。  
5) **验收与回滚**：可灰度开关（Mock/真实可切换），并能一键回到 Mock。

### 1.3 非目标（本期不做）
- 语音对话（语音→语音）Realtime 大模型链路  
- 多语言自动识别/自动翻译  
- 高级对齐（语音时间轴逐字高亮）  
- 深度“修订 diff 对比”面板（本期只保证修订可感知）

---

## 2. 用户与场景

### 2.1 核心场景
- 用户打开插件 → 进入 AI 语音笔记 → 开始录音  
- 录音中实时出字（partial）并逐步落盘为 final  
- 结束后进入编辑态，可继续补录或保存到墨问

### 2.2 边界场景（必须处理）
- 麦克风权限拒绝/设备不可用  
- 处在 chrome://、扩展页、空白页等不可录音场景（前端已限制提示）  
- 网络抖动 / WS 断开 / 火山返回错误  
- 长时间录音（> 5~30min）造成连接/内存压力

## 3. 总体方案（架构与数据流）

### 3.1 总体架构（推荐：后端中转）
**Browser Extension (React)**  
↕（WebSocket/SSE/HTTP）  
**你的中转服务（Proxy/Relay）**  
↕（WebSocket）  
**火山引擎流式 ASR**

**为什么必须中转**  
- 扩展端不能暴露火山 AppKey/Token（一旦泄露可被滥用、超额计费）  
- 中转可做：鉴权、限流、审计、重连、协议适配、日志采样、灰度开关

### 3.2 数据流（端到端）
1) 前端获取麦克风音频流（PCM/Opus 以你实现为准）  
2) 前端分包（推荐 100–200ms 一包）发送到中转服务  
3) 中转服务按火山 WS 协议推送音频帧  
4) 火山返回中间/最终识别结果  
5) 中转服务把结果转换成统一事件（partial/final/revise）推回前端  
6) 前端按既有 UI 机制渲染与编辑  
7) 用户保存 → 复用现有墨问 create note 流程

## 4. 功能需求（FR）

### 4.1 语音识别链路开关
- FR-1：新增配置 `asrProvider = mock | volcengine`（默认 mock，可灰度）  
- FR-2：当 `volcengine` 启用时，UI 与用户流程不变化，仅数据源替换  
- FR-3：提供“快速自检”入口（开发/测试用）：显示当前 ASR Provider、连接状态、最近 5 条事件

### 4.2 前端采集与推流
- FR-4：采集音频满足火山要求的采样率/声道/位深（以火山文档为准）  
- FR-5：前端分包推送：
  - 每包时长：100–200ms（可配置）
  - 支持 pause：暂停时停止推送，但保留会话；继续后恢复
- FR-6：录音结束：发送“结束帧/结束信号”（按火山协议）并进入 finalizing

### 4.3 结果映射（关键：不改 UI）
你现有 UI 事件模型：
- `partial(fullPartialText)`
- `final(appendText)`
- `revise(targetBlockIndex, newText)`（可选）

火山结果 → 统一事件的映射策略（以“能稳定跑通”为优先）：

**基础映射（本期必做）**
- M-1：火山返回的“当前识别中间态” → `partial(text)`（覆盖更新）
- M-2：火山返回的“最终结果片段” → `final(text)`（追加到最后一个 block 或新 block）

**修订映射（本期建议做，做不到也不阻塞上线）**
- M-3：若火山返回包含“同一句被改写/替换”的标识（或你能检测到 final 的回改）：
  - 触发 `revise(blockIndex, newText)`
- M-4：若火山没有 revise 概念，则先用“final 追加 + 本地轻量修订策略”模拟：
  - 当 final 片段与上一段尾部重叠度高，认为是回改 → 用 `revise` 替换上一段（详见 6.3）

> 结论：**UI 保持 Typeless**，即使后端不提供 revise，也能靠“重叠检测”实现“修订可感知”。

### 4.4 段落切分与可读性
- FR-7：支持标点、ITN 等“可读性增强”开关（由中转服务配置）  
- FR-8：段落策略：
  - 若识别结果本身含 `\n\n`：按空行切段
  - 否则：按中文标点（。！？；）+ 字数阈值软分段（阈值可配置）
- FR-9：结束整理（finalizing）：
  - 合并最终 final + 残留 partial
  - 做一次“收尾处理”：去除重复空格、连续换行归一（不做重写润色）

### 4.5 错误处理与可恢复
- FR-10：连接失败/鉴权失败：UI 显示清晰错误 + 提供重试  
- FR-11：WS 中断：自动重连（限制次数），重连失败则回到 editing（保留已转写文本）  
- FR-12：服务端限流/超额：提示用户稍后再试（不清空文本）  
- FR-13：任何异常都必须保证：**已得到的 final 文本不丢失**

### 4.6 日志与可观测（上线前必须）
- FR-14：记录每次会话关键指标（不含敏感音频原文，默认不开启音频存储）：
  - sessionId、开始时间、结束时间、总时长
  - 首字延迟（TTFB）
  - partial 更新次数
  - final 片段次数
  - 重连次数、错误码
- FR-15：支持“采样级别”日志（可开关）：
  - 采样保存最后 N 条 ASR 事件（便于复现）
  - 仅在开发/灰度环境开启

## 5. 非功能需求（NFR）

### 5.1 性能
- NFR-1：首字延迟（TTFB）目标：<= 1.0–1.5s（视网络与模型而定）  
- NFR-2：partial 更新频率：200–500ms 体感平滑  
- NFR-3：前端渲染不卡顿（长文本按段渲染，必要时虚拟列表后续再上）

### 5.2 安全
- NFR-4：火山密钥仅存于服务端环境变量/密钥管理，不下发前端  
- NFR-5：中转服务对每个用户/设备做速率限制与并发限制  
- NFR-6：对请求做审计：sessionId + userId（或匿名 deviceId）

### 5.3 可靠性
- NFR-7：中转服务支持无状态扩容（尽量把会话状态放内存 + 可选 Redis）  
- NFR-8：故障降级：可切回 mock，不阻塞语音笔记整体功能

## 6. 详细设计（可直接给研发实现）

### 6.1 中转服务（Proxy/Relay）职责
1) **鉴权层**：
   - 校验用户（或匿名设备）是否允许使用 ASR
   - 生成内部 `sessionId`
2) **连接火山**：
   - 按火山 WS 协议建立连接
   - 注入火山所需鉴权参数（按文档）
3) **音频转发**：
   - 接收前端音频帧
   - 按火山协议封装并发送
4) **结果适配**：
   - 接收火山识别结果
   - 转换为统一事件：partial/final/revise
   - 推送给前端
5) **重连与错误处理**：
   - 网络中断/火山断开：按策略重连或结束
6) **指标与日志**：
   - 记录 TTFB、错误码、吞吐等

### 6.2 前端与中转的协议（建议）
建议你内部定义一个**稳定的前后端协议**，不让前端直接依赖火山字段。

**前端 → 中转（WebSocket）**
- `start`：{ sessionId?, audioConfig, asrConfig }
- `audio`：{ seq, ts, payload(base64/arraybuffer) }
- `pause` / `resume`
- `stop`

**中转 → 前端**
- `ready`：{ sessionId }
- `event_partial`：{ text }
- `event_final`：{ text }
- `event_revise`：{ blockIndex, text }
- `error`：{ code, message, retriable }
- `metrics`：{ ttfbMs, partialCount, finalCount }

> 备注：payload 建议走二进制帧（ArrayBuffer）减少开销；base64 仅用于开发。

### 6.3 “修订(revise)”的落地策略（你可以直接照做）
若火山本身无 revise 事件，本期用“重叠检测 + 替换上一段”模拟：

- 保存 `lastFinalTail`（上一段尾部 N 字，如 30–60）
- 当收到新的 final 片段 `newFinal` 时：
  - 计算 `newFinal` 与 `lastFinalTail` 的最大重叠（suffix/prefix overlap）
  - 若重叠长度 >= 阈值（如 12–18 字），判定为回改：
    - 用 `revise(lastBlockIndex, oldBlockText + newFinal[overlap:])`
  - 否则：
    - 普通追加 `final(newFinal)`

同时触发 UI 的“已修订”提示（你已实现高亮+提示）。

### 6.4 ASR 参数（按火山文档可配置）
中转服务提供配置项（存配置中心/环境变量），并暴露给前端只读（用于调试展示）：
- `enablePunc`：标点
- `enableITN`：数字规整
- `enableSpeaker`：说话人（本期默认关）
- `hotwords`：热词（本期可先给静态列表：墨问、MCP、Readability…）

> 具体字段名称与取值必须以火山官方文档为准；本 PRD 不硬编码字段名，避免文档版本差异。

### 9.3 安全测试

## 12. 验收标准（必须全部通过）
- A1：真实 ASR 下，录音开始后 1–1.5s 内开始吐字（视网络可略放宽）
