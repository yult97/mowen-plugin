摘要

P2

ZIP 导出的 PDF 丢了兼容性归一化

重构后，单篇导出和合并导出仍然会清理 PDF 的 OpenAction / ViewerPreferences， 但 ZIP 批量导出不再清理，导致同一功能下三条导出链路行为不一致。

P3

工具栏清理选择器过宽，可能删掉站点自身节点

现在会删除页面上所有类名为 .mowen-selection-toolbar 的元素。 如果宿主网页刚好用了同名 class，第一次划线时就会误删对方节点。

P2

调整 1：恢复 ZIP 导出的 PDF 兼容性归一化

文件：src/utils/pdfExporter.ts

为什么这里要改

  • 历史行为里，所有导出产物都经过兼容性归一化。
  • 拆分后，generatePdfBlob() 改成直接调用 outputPdfBlob(worker)。
  • 这个函数默认不会做归一化。
  • 而 exportBatchAsZip() 正是靠 generatePdfBlob() 来生成每个 PDF 文件。
  • 结果就是：单篇导出正常、合并导出正常、ZIP 里的 PDF 回退成未归一化版本。
受影响流程
Popup 单篇导出 仍然正常，因为走的是 exportSinglePdf()，那里还保留了归一化。
NotesExport 合并导出 仍然正常，因为走的是 exportMergedPdf()。
NotesExport 批量 ZIP 导出 这里发生回归，因为 ZIP 里的每个 PDF 来自 generatePdfBlob()。

建议怎么调

当前代码
const blob = await outputPdfBlob(worker);
建议改成
const blob = await outputPdfBlob(worker, {
  normalizeForViewer: true,
});
如果你的目标是“所有导出产物保持同一兼容策略”，这里最直接的调整就是把 generatePdfBlob() 也接回归一化流程。
P3

调整 2：只清理扩展自己创建的工具栏节点

文件：src/content/highlighter/SelectionToolbar.ts

为什么这里要改

  • 现在的清理逻辑会执行 document.querySelectorAll('.mowen-selection-toolbar')。
  • 这个选择器只看 class，不区分“是扩展自己的 DOM”还是“网页本身的 DOM”。
  • 如果宿主页面恰好也用了同名 class，第一次划线时会把对方元素一起删掉。
  • 这不是理论上的代码洁癖问题，而是会直接改坏网页本身的显示结构。
当前代码方向
document.querySelectorAll('.mowen-selection-toolbar').forEach((node) => {
  if (node instanceof HTMLElement) {
    node.remove();
  }
});

建议怎么调

  • 给扩展创建的节点加一个专属标记，例如 data-mowen-owned="true"。
  • 清理时改成只匹配这个标记，不要匹配所有同名 class。
  • 如果只是想移除旧实例，也可以优先直接处理 this.container 或缓存的旧引用。
建议代码方向
document
  .querySelectorAll('.mowen-selection-toolbar[data-mowen-owned="true"]')
  .forEach((node) => {
    if (node instanceof HTMLElement) {
      node.remove();
    }
  });

this.container = document.createElement('div');
this.container.className = 'mowen-selection-toolbar';
this.container.dataset.mowenOwned = 'true';
这个改法的核心不是“换个 selector 写法”，而是把“扩展 DOM”和“宿主页面 DOM”做边界隔离。
建议落地顺序：先修 src/utils/pdfExporter.ts 的 ZIP 导出兼容性，再修 src/content/highlighter/SelectionToolbar.ts 的 DOM 清理边界。 如果你要，我下一步可以直接把这两处代码一起改掉并补一轮构建验证。
